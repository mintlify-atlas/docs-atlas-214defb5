Xget is a Cloudflare Workers project developed with Wrangler. This page covers setting up a local environment, running the test suite, and contributing changes. The local dev server simulates the Cloudflare Workers runtime on your machine, so you can test request routing, caching behavior, and protocol handling without deploying.

## Prerequisites

Before you start, make sure you have the following installed:

- **Node.js 24** and **npm**
- **Wrangler CLI** — installed automatically via `npm install`, or globally with `npm install -g wrangler`
- A **Cloudflare account** — required only for deploying; not needed for local development or testing

## Setup

<Steps>
  <Step title="Fork and clone the repository">
    Fork [xixu-me/xget](https://github.com/xixu-me/xget) on GitHub, then clone your fork:

    ```bash
    git clone https://github.com/your-username/xget.git
    cd xget
    ```
  </Step>
  <Step title="Install dependencies">
    ```bash
    npm install
    ```

    This installs Wrangler, Vitest, ESLint, Prettier, TypeScript, and all other dev dependencies.
  </Step>
  <Step title="Authenticate with Cloudflare (first time only)">
    This step is only required if you want to deploy to Cloudflare Workers. Local development and testing work without it.

    ```bash
    npx wrangler login
    ```

    This opens a browser window to authenticate with your Cloudflare account.
  </Step>
  <Step title="Start the dev server">
    ```bash
    npm run dev
    ```

    Wrangler starts a local dev server at `http://localhost:8787`. Requests are handled by the full Worker pipeline — including platform routing, caching, retry logic, and security headers.
  </Step>
</Steps>

<Tip>
  Use `npm run dev` to test URL routing and protocol handling locally. The Wrangler dev server runs the same Workers runtime that Cloudflare uses in production, so behavior is consistent between local and deployed environments.
</Tip>

## Available scripts

| Script | Description |
|---|---|
| `npm run dev` / `npm start` | Start the Wrangler dev server at `http://localhost:8787` |
| `npm run test:run` | Run the full test suite once |
| `npm run test:watch` | Run tests in watch mode (re-runs on file changes) |
| `npm run test:coverage` | Run tests and generate an Istanbul coverage report |
| `npm run test:ui` | Open the Vitest browser UI |
| `npm run lint` | Run ESLint on `src/`, `test/`, and `adapters/` |
| `npm run lint:fix` | Run ESLint with auto-fix |
| `npm run format` | Format all source files with Prettier |
| `npm run format:check` | Check formatting without writing changes |
| `npm run type-check` | Run TypeScript type checking with `tsc --noEmit` |
| `npm run deploy` | Deploy to Cloudflare Workers production |

## Testing

Xget uses [Vitest](https://vitest.dev/) with `@cloudflare/vitest-pool-workers` to run tests inside the Workers runtime. Coverage is generated via `@vitest/coverage-istanbul`.

**Run all tests once:**

```bash
npm run test:run
```

**Generate a coverage report:**

```bash
npm run test:coverage
```

The test suite is organized into four categories:

- **Unit tests** — isolated tests for individual utility functions, config parsing, path transformers, and routing helpers
- **Integration tests** — end-to-end tests that exercise the full request pipeline for each platform
- **Security tests** — validate that security headers are present, methods are enforced, and path length limits are respected
- **Performance tests** — verify that the `PerformanceMonitor` produces correct timing marks and that `X-Performance-Metrics` headers are set

Tests live in the `test/` directory. Add or update tests whenever you change behavior that affects routing, headers, cache logic, retries, security controls, or protocol compatibility.

## Project structure

```
xget/
├── src/
│   ├── app/          # Main request handler and request context
│   ├── config/       # Platform catalog (PLATFORM_CATALOG) and app config (CONFIG)
│   ├── protocols/    # Git, Docker, AI, and Hugging Face protocol handlers
│   ├── routing/      # Platform matching, path transformation, and target resolution
│   ├── upstream/     # Cache lookup, upstream fetch, and retry logic
│   ├── response/     # Response finalization and header processing
│   └── utils/        # Security headers, CORS, request validation, performance monitoring
├── adapters/         # Runtime adapters for Vercel, Netlify, and Deno Deploy
├── test/             # Full test suite (unit, integration, security, performance)
├── wrangler.toml     # Cloudflare Workers configuration
└── package.json
```

Each directory has a single clear responsibility. When adding a new platform, you typically only need to touch `src/config/platform-catalog.js` and optionally `src/routing/platform-transformers.js`. Changes to protocol-level behavior go in `src/protocols/`.

## Contributing

Xget welcomes contributions of all sizes, including bug reports, documentation improvements, test coverage, and new platform support.

### Before you open a pull request

Run the following checks locally and ensure they all pass:

```bash
npm run lint
npm run format:check
npm run test:run
npm run type-check
```

### Commit message format

This repository uses [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/). All commit messages must follow the format:

```text
type(scope): description
```

Examples:

```text
feat(docker): normalize blob redirect handling
fix(routing): preserve crates search queries
docs(readme): clarify npm registry setup
```

Commit message validation is enforced automatically via `commitlint` on the `commit-msg` git hook.

### Pull request workflow

1. Fork the repository and create a branch from `main`.
2. Keep your change focused — avoid mixing unrelated fixes in a single PR.
3. Add or update tests when behavior changes.
4. Update documentation when user-facing behavior, configuration, or platform support changes.
5. Open a pull request using the repository template and describe the user impact clearly.

### Code style

- ESLint enforces code style rules (`npm run lint`).
- Prettier enforces formatting (`npm run format:check`).
- Follow the existing project structure and naming conventions.
- Avoid logging secrets, tokens, or private request data.
- Document new platform prefixes and examples in `README.md` when adding support for a new platform.
